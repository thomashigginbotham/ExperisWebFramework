HTML and CSS Style Guide

The HTML and CSS Style Guide is an attempt to document the standards to which our markup and styles will be written, thereby improving code quality and collaboration. These guidelines only apply to source files. Final CSS should be compressed/minified.

HTML5 Polyglot Markup

For maximum compatibility with content management systems and HTML/XHTML parsers, we recommend adhering to the polyglot markup standards, as outlined by the W3C.

General Rules

  • Use HTML5 sectioning elements appropriately (<header/>, <footer/>, <section/>, <aside/>, etc.)
  • Use ID and class names that are as short as possible but long as necessary for comprehension.
  • Do not over-qualify selectors (e.g. use .my-class, not ul.my-class).
  • Do not use more than one ID selector in a rule.
  • Use shorthand properties (margin, font, background, etc.) when possible.
  • Instead of using multiple class names to style an element, consider combining them into one class (e.g. instead of .item.small, use .small-item).
  • Use relative protocols in URLs unless a secure protocol is unavailable (e.g. background-image: url(//somedomain.suffix/image.jpg)).
  • Do not use double quotes, and only use single quotes when necessary (e.g. font: 1.2em 'Times New Roman' is fine, but not background: url('my-image.jpg')).
  • Avoid CSS “hacks.” Search for another solution first.

HTML Coding Style

  • Use one tab character for each level of indentation. Space characters should not be used for indenting. If your HTML editor supports the display of white space characters, turn it on to verify that spaces and tabs are being used appropriately.
  • In accordance with polyglot markup, all tags should be lowercase, all attributes should be contained in double quotes, and empty elements, such as <img/> and <br/> should be closed.

Example syntax:

<!doctype html>
<html xmlns="http://www.w3.org/1999/xhtml" lang="en-us" xml:lang="en-us">
	<head>
		<meta charset="utf-8"/>
		<title>Sample Title</title>
	</head>

	<body>
		<header>
			<hgroup>…</hgroup>
		</header>

		<nav>…</nav>

		<section id="main">
			<article>…</article>
			<aside>…</aside>
		</section>

		<footer>…</footer>
	</body>
</html>

CSS Coding Style

  • CSS documents should have a table of contents at the top, and major sections should be prefixed with noticeable comments.
  • Use one tab character for each level of indentation. Space characters should not be used for indenting. If your CSS editor supports the display of white space characters, turn it on to verify that spaces and tabs are being used appropriately.
  • Use one space character after each : in property declarations.
  • Use one space character before each { in rule declarations.
  • Use lowercase hex color codes.
  • Properties should be grouped by type (block styles, colors, fonts, etc.)

Example syntax (notice how similar properties are grouped together):

.sample-class {
	display: block;
	margin: 0 auto;
	padding: 1em;
	width: 50%;
	min-height: 2em;
	border: 1px solid #fc0;
	color: #333;
	background-color: #fff;
	font-size: 1.2em;
	font-weight: bold;
}

SCSS Style

  • Any $variable or @mixin used in more than one file should be moved into a global file.
  • Variables and mixins should appear at the top of the CSS document, after the TOC.
  • Avoid nesting more than three levels deep if possible.

SASS/CSS File Organization

CSS file structure should be as follows:

  • css
    • pages
      • home.css
      • interior.css
  • scss
    • components
      • _forms.scss
    • global
      • _framework.scss
      • _site.scss
      • _variables.scss
    • pages
      • home.scss
      • interior.scss
    • plugins
      • _jquery.some-plugin.scss
      • _nice-slider.scss

As you can see, the final, minified CSS will only consist of files from the pages directory. Each file in the pages directory will import files that are needed from the other directories. See the SASS reference for details on importing partials.

Units of Measurement

  • Relative units, such as em and %, should be used instead of non-relative units, such as px, when possible.
  • The line-height property should always be unitless.
  • Omit unit specification after “0” values.

Specificity

Use ID selectors sparingly and only for unique areas of a website that will never be repeated. If there is a doubt, use a class selector instead.

Final Words

Above all, be consistent! The purpose of a style guide is to make collaboration reliable and easy. If you are unsure how to format something, check other sections or files to see how previous developers handled the formatting. Without consistency, your team may be thrown as to what your intentions were instead of focusing on writing code.
